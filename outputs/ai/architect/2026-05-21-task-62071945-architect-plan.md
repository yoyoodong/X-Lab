# 产品虾：选题判断与调度验证

**记录日期：** 2026-05-21
**来源任务：** 飞书任务「测试 X_Lab 飞书任务入口：生成今日虾团队简报」
**当前环节：** 产品虾调度响应（流程验证）

---

## 一、任务理解

本轮不是一个真实的产品策划任务。

本质是：飞书任务入口触发 → 虾老大拆任务 → 虾老大调度到产品虾 → 产品虾能否正确识别指令并给出产品视角的判断。

虾老大已经把任务定性为**内容类任务（简报）**，按调度规则，产品虾在内容类任务中承担**选题策划**环节。但本次是测试，没有真实情报、没有读者需求、没有发布目标，所以我不会编造选题。

我的职责边界是：
- 判断「简报」作为一个产品，它的目标用户是谁、解决什么问题
- 给出简报选题的框架（不是具体选题本身）
- 表达产品虾在调度链中能提供什么价值
- 验证产品虾从被调度到产出判断这一段链路是否正常

---

## 二、我的判断：简报作为信息产品的定位

即使没有今日具体情报，我仍然可以从产品角度给「虾团队简报」做一次定位审视：

| 产品维度 | 当前可用信息 | 产品虾判断 |
|---|---|---|
| 目标用户 | 当前资料未提供 | 推测为羊（决策者）或团队内部，需要补充 |
| 用户痛点 | 当前资料未提供 | 若团队信息分散在多个虾各自产出中，则简报的痛点是「汇总成本高」，需要统一视图 |
| 使用场景 | 当前资料未提供 | 推测为每日快速浏览（3 分钟内了解团队全貌），非深度分析 |
| 核心价值 | 当前资料未提供 | 信息聚合 + 优先级排序 + 风险暴露 |
| 竞品参照 | 当前资料未提供 | 可参考 Notion 日报模板、Slack Standup Bot 逻辑 |
| MVP 范围 | 当前资料未提供 | 状态表 + 情报摘要 + 风险标记，不需要精美排版 |

### 功能优先级建议（初步）

| 优先级 | 功能要素 | 理由 |
|---|---|---|
| P0 | 各虾状态（在线/离线/阻塞） | 无状态则无法调度 |
| P0 | 今日关键情报（≤3 条） | 简报的核心信息价值 |
| P1 | 进行中任务 + 进度 | 跟踪任务流转 |
| P1 | 风险与阻塞标记 | 提前暴露问题 |
| P2 | 设计美化 | 先通后美 |
| P3 | 历史趋势/对比 | 稳定运行后再加 |

这些判断在当前测试任务中无法填充真实内容，但它们构成了「简报」这个信息产品的基本骨架。

## 三、本轮交付内容

### 3.1 产品虾调度响应确认

| 检查项 | 结果 |
|---|---|
| 是否识别到任务类型 | ✅ 识别为内容类任务（简报） |
| 是否理解产品虾在调度链的位置 | ✅ 内容类链路：产品虾负责选题策划，位于虾老大之后、运营虾之前 |
| 是否产出产品角度判断 | ✅ 已产出简报产品定位、功能优先级（框架层） |
| 是否编造数据 | ❌ 未编造任何情报、进度、外部信息 |
| 是否标记缺失资料 | ✅ 已标记「当前资料未提供」的字段 |

### 3.2 为下一环节提供的输入

如果这是正式任务、情报已到位，产品虾会给运营虾这样的选题 brief：

```text
【简报选题 brief】
- 主题：今日虾团队简报
- 目标用户：羊（决策者）
- 核心信息：①团队各角色状态 ②今日关键情报（侦察虾提供） ③风险标记
- 调性：简洁、面向决策、不煽情
- 格式要求：状态表 + 3 条情报摘要 + 1 条风险（如有）
- 字数预期：300 字以内
- 不需要：行业新闻列表、长段分析、营销语气
```

但由于当前是测试任务，这个 brief 仅作为格式示意，供运营虾参考链路。

## 四、风险与注意事项

| 风险条目 | 说明 |
|---|---|
| 内容空值风险 | 简报定位和功能优先级是在没有真实情报的情况下做的框架判断。真实启动时需侦察虾先提供情报，产品虾才能产出具体选题。 |
| 调度链断裂风险 | 本轮链路为「虾老大 → 产品虾」。如果产品虾产出后没有明确指定下一个角色，链路会中断。需要虾老大确认调度方向后，产品虾传递 brief 给运营虾。 |
| 用户画像缺失风险 | 简报的读者是谁（羊？团队成员？外部关注者？）会直接决定简报的选题和深度。当前资料未提供，需要补充。 |
| 过度设计风险 | 测试阶段不宜把简报定位做得太复杂。建议先跑通状态表 + 情报摘要的最小闭环，后续再迭代。 |
